# Terrabia – Web & Mobile Application Context

Terrabia is a multi-role platform that connects **agricultors (farmers)**, **customers**, **delivery agencies**, and **administrators**.

This document describes:

- The **business rules**
- The **user roles and flows**
- The **web application pages and URLs**
- The **backend data model and endpoints** (at a high level)

It is meant to be the contract between the **frontend team (mobile + web)** and the **Django backend team**.

---

## 1. Tech Stack & Architecture

### 1.1 Frontend (Web)

- HTML, CSS, JavaScript (no heavy SPA requirement; progressive enhancement is OK).
- Django templates for server-side rendered pages.
- Optional AJAX/JSON endpoints for dynamic actions:
  - Cart updates
  - Messaging
  - Status changes

Mobile frontend (existing) communicates with the same backend via JSON APIs.

### 1.2 Backend

- Django project (e.g. `terrabia`) with multiple apps (suggested):
  - `accounts` – authentication, user profiles, roles.
  - `marketplace` – products, categories, farmer-facing features.
  - `orders` – cart, orders, payments.
  - `delivery` – agencies, deliveries, tracking.
  - `messaging` – conversations & messages.
  - `platform_admin` – admin dashboards & management UI (not Django admin).

- Authentication:
  - **Web**: Django session auth.
  - **Mobile**: token/JWT-based API (if needed later).

- Shared domain model for both web and mobile.

## 2. User Roles & Rules

### 2.1 Roles

1. **Customer**
   - Signs up & logs in through the app.
   - Buys products from farmers.
   - Manages cart, orders, and profile.

2. **Farmer (Agricultor)**
   - Signs up & logs in through the app.
   - Manages products and stock.
   - Receives and manages orders.
   - Collaborates with delivery agencies.
   - Uses messaging for communication.

3. **Delivery Agency**
   - **Cannot sign up**.
   - Created by **Administrator**.
   - Gets email + default password generated by the system.
   - Logs in using these credentials.
   - Manages assigned deliveries and messaging.

4. **Administrator**
   - **Cannot sign up in the app**.
   - Accounts are created outside (e.g. fixtures, superuser, or internal tool).
   - Logs in via the same login form.
   - Manages users and agencies and monitors platform metrics.

### 2.2 Access Rules

- Customers and Farmers can see only their own data (profile, orders, products, etc.).
- Delivery agencies see only their assigned deliveries and conversations.
- Administrators have full access to:
  - User management.
  - Delivery agencies management.
  - High-level stats and platform-wide metrics.

Frontend must **hide** UI actions that the role cannot perform, but backend must still enforce permissions.

## 3. Core Features by Role

### 3.1 Customer

- Browse products and farmers.
- Search and filter products by category/location.
- Add products to cart and manage cart.
- Checkout and pay (Mobile Money / Card).
- View orders and track delivery status.
- Manage profile and settings (address, contact info).

### 3.2 Farmer

- Dashboard with sales/metrics and recent orders.
- Manage products (CRUD).
- View orders (per product and per customer).
- Contact delivery agencies for orders.
- Messaging with agencies (and optionally customers).
- Manage farm profile and settings.

### 3.3 Delivery Agency

- Dashboard with metrics and recent deliveries.
- View assigned deliveries by status (Pending, In Progress, Completed).
- Update delivery status.
- Messaging with farmers/customers.
- Manage agency profile, documents, and credentials.

### 3.4 Administrator

- Platform overview dashboard:
  - Totals (users, farmers, agencies, orders, revenue).
  - Trend charts (revenue, orders).
  - Recent activities.
- User management (customers & farmers).
- Delivery agency management:
  - Create agencies and auto-generate credentials.
  - Verify / reject agencies.
  - Resend credentials.
- Admin profile and settings.

## 4. URL & Page Map (Web)

> **Note:** Paths are indicative. Backend can adjust, but the final routes must stay consistent with what frontend uses.

### 4.1 Public & Auth

1. `/`
   - Landing page: description of Terrabia, CTA buttons.
   - Actions:
     - "Get started" → `/join`
     - "Sign In" → `/auth/login`

2. `/join`
   - Role selection:
     - "I am a Customer" → `/auth/register/customer`
     - "I am a Farmer" → `/auth/register/farmer`
   - Link: "Already have an account? Sign In" → `/auth/login`

3. `/auth/register/customer`
   - Customer signup form:
     - Full name
     - Email
     - Phone
     - Address
     - Password + Confirm password
   - On success → redirect to customer home: `/customer/home`.

4. `/auth/register/farmer`
   - Farmer signup form:
     - Farm name
     - Contact person name
     - Email
     - Phone
     - Farm address
     - Password + Confirm password
   - On success → redirect to farmer dashboard: `/farmer/dashboard`.

5. `/auth/login`
   - Shared sign-in for all roles:
     - Email
     - Password
   - After login:
     - `role=customer` → `/customer/home`
     - `role=farmer` → `/farmer/dashboard`
     - `role=admin` → `/admin/dashboard`
     - `role=delivery_agency` → `/agency/dashboard`

6. `/auth/logout`
   - Logs out user and redirects to `/`.

7. `/auth/forgot-password` (if implemented later)
   - Form: email.
   - Sends reset link / code.

## 5. Core Data Model (Backend / Django)

> These are conceptual models. Backend can adapt but must preserve fields needed by the UI.

### 5.1 Users & Profiles

```python
# accounts/models.py
from django.contrib.auth.models import AbstractUser
from django.db import models

class User(AbstractUser):
    ROLE_CUSTOMER = "customer"
    ROLE_FARMER = "farmer"
    ROLE_ADMIN = "admin"
    ROLE_AGENCY = "delivery_agency"

    ROLE_CHOICES = [
        (ROLE_CUSTOMER, "Customer"),
        (ROLE_FARMER, "Farmer"),
        (ROLE_ADMIN, "Administrator"),
        (ROLE_AGENCY, "Delivery agency"),
    ]

    role = models.CharField(max_length=20, choices=ROLE_CHOICES)

class CustomerProfile(models.Model):
    user = models.OneToOneField(User, on_delete=models.CASCADE)
    full_name = models.CharField(max_length=120)
    phone = models.CharField(max_length=30)
    address = models.TextField(blank=True)

class FarmerProfile(models.Model):
    user = models.OneToOneField(User, on_delete=models.CASCADE)
    farm_name = models.CharField(max_length=120)
    contact_person = models.CharField(max_length=120)
    phone = models.CharField(max_length=30)
    address = models.TextField()

class DeliveryAgency(models.Model):
    user = models.OneToOneField(User, on_delete=models.CASCADE)
    name = models.CharField(max_length=120)
    coverage_area = models.CharField(max_length=255)
    license_number = models.CharField(max_length=50)
    STATUS_PENDING = "pending"
    STATUS_VERIFIED = "verified"
    STATUS_REJECTED = "rejected"
    status = models.CharField(
        max_length=20,
        choices=[
            (STATUS_PENDING, "Pending"),
            (STATUS_VERIFIED, "Verified"),
            (STATUS_REJECTED, "Rejected"),
        ],
        default=STATUS_PENDING,
    )
    member_since = models.DateTimeField(auto_now_add=True)
```

### 5.3 Cart & Orders

```python
# orders/models.py
class CartItem(models.Model):
    customer = models.ForeignKey(CustomerProfile, on_delete=models.CASCADE)
    product = models.ForeignKey(Product, on_delete=models.CASCADE)
    quantity = models.DecimalField(max_digits=10, decimal_places=2)
    created_at = models.DateTimeField(auto_now_add=True)

class Order(models.Model):
    STATUS_PROCESSING = "processing"
    STATUS_SHIPPED = "shipped"
    STATUS_DELIVERED = "delivered"
    STATUS_PENDING = "pending"

    status = models.CharField(max_length=20, default=STATUS_PROCESSING)
    customer = models.ForeignKey(CustomerProfile, on_delete=models.CASCADE)
    code = models.CharField(max_length=20, unique=True)  # CMD001
    created_at = models.DateTimeField(auto_now_add=True)
    delivery_address = models.TextField()
    total_amount = models.DecimalField(max_digits=12, decimal_places=2)

class OrderItem(models.Model):
    order = models.ForeignKey(Order, on_delete=models.CASCADE, related_name="items")
    product = models.ForeignKey(Product, on_delete=models.PROTECT)
    farmer = models.ForeignKey(FarmerProfile, on_delete=models.PROTECT)
    quantity = models.DecimalField(max_digits=10, decimal_places=2)
    unit_price = models.DecimalField(max_digits=10, decimal_places=2)
    line_total = models.DecimalField(max_digits=12, decimal_places=2)
```

## 6. API & AJAX Endpoints (for Web + Mobile)

> Exact URLs/methods can be adjusted, but frontend expects endpoints with similar responsibilities.

### 6.1 Auth

- `POST /api/auth/login/` → `{ email, password }` → `{ user, token (for mobile) }`
- `POST /api/auth/register/customer/`
- `POST /api/auth/register/farmer/`
- `POST /api/auth/logout/`
- `POST /api/auth/forgot-password/`

> **Note:** Web may use classic Django views with forms; mobile uses `/api/...` JSON endpoints.

### 6.2 Products

- `GET /api/products/` – filters: `?search=&category=&location=`
- `GET /api/products/<id>/`
- `GET /api/farmer/products/` (farmer's own products)
- `POST /api/farmer/products/`
- `PUT/PATCH /api/farmer/products/<id>/`
- `DELETE /api/farmer/products/<id>/`

### 6.3 Cart & Orders (Customer)

- `GET /api/cart/`
- `POST /api/cart/items/`
- `PATCH /api/cart/items/<id>/` – update quantity
- `DELETE /api/cart/items/<id>/`
- `POST /api/orders/` – from cart + delivery address + payment method
- `GET /api/orders/` – list for logged-in customer
- `GET /api/orders/<id>/` – detail + timeline + agency

### 6.4 Farmer Orders

- `GET /api/farmer/orders/`
- `GET /api/farmer/orders/<id>/`
- Optional: `POST /api/farmer/orders/<id>/mark-ready/`

### 6.5 Delivery (Agency)

- `GET /api/agency/deliveries/`
- `GET /api/agency/deliveries/<id>/`
- `PATCH /api/agency/deliveries/<id>/` – update status

### 6.6 Messaging

- `GET /api/messages/threads/` – list threads for current user
- `GET /api/messages/threads/<id>/` – messages in thread
- `POST /api/messages/threads/` – create (if needed)
- `POST /api/messages/threads/<id>/messages/` – send message
- Optional: websockets for real-time

### 6.7 Admin – Agencies

- `GET /api/admin/agencies/`
- `POST /api/admin/agencies/` – create agency and generate credentials
- `POST /api/admin/agencies/<id>/verify/`
- `POST /api/admin/agencies/<id>/reject/`
- `POST /api/admin/agencies/<id>/resend-credentials/`

## 7. Authentication & Authorization Logic

Use Django's User model (custom) with role field.

On every view/API:

- Check `request.user.is_authenticated`
- Check `request.user.role` for access

**Example restrictions:**

- `/customer/*` → `user.role == "customer"`
- `/farmer/*` → `user.role == "farmer"`
- `/agency/*` → `user.role == "delivery_agency"`
- `/admin/*` → `user.role == "admin"`

## 8. Non-Functional Notes

### 8.1 Security

- Strong password validation
- CSRF protection on forms & AJAX
- Rate limiting for login if possible
- Server-side checks for all sensitive actions

### 8.2 Internationalization

- Language toggles could be implemented later
- UI text should be defined in templates to allow translations

### 8.3 Performance

- Pagination for lists (products, orders, agencies, messages)
- Cache read-heavy lists (e.g. product catalog, categories)
